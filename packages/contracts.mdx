`@jami-studio/harness-contracts` is the machine-readable foundation of Jami Harness. It defines JSON Schema anchors and compatibility fixtures for every data structure the harness may emit or consume at the harness/Studio UI boundary — covering policy decisions, tool executions, memory records, evidence packets, and more. Nothing in this package owns Studio UI primitives, renderer implementation, or token outputs; it is purely a contract layer that both sides of the boundary read from.

## Installation

```bash
npm install @jami-studio/harness-contracts
```

**Package:** `@jami-studio/harness-contracts` · **Version:** `0.1.0`

### Consuming schemas in TypeScript

Import the generated `contracts.ts` export to access schema IDs and anchor metadata directly in your validator setup:

```ts
import contracts from '@jami-studio/harness-contracts/generated/contracts';

// Access a schema ID for use with your JSON Schema validator
console.log(contracts.runEvent.id);
// → 'https://jami.studio/schemas/harness/run-event.schema.json'

console.log(contracts.toolExecution.id);
// → 'https://jami.studio/schemas/harness/tool-execution.schema.json'
```

### Consuming schemas via OpenAPI

If you prefer OpenAPI `$ref` resolution, import `openapi.json` into your document:

```json
{
  "$ref": "node_modules/@jami-studio/harness-contracts/generated/openapi.json#/components/schemas/PolicyDecision"
}
```

### Schema IDs are the stable reference

Always reference anchors by their schema ID, not by file path. The schema IDs are stable across patch releases and are what your validators and OpenAPI tooling should resolve against.

## Cross-Field Validation Rules

The contracts validation gate enforces semantic rules that go beyond structural JSON Schema checks. Your fixtures and consuming code must satisfy all of these:

<Steps>
  <Step title="Non-allow policy decisions carry audit evidence">
    Every `policyDecision` with an outcome other than `allow` must include an audit evidence reference. Decisions without evidence are rejected.
  </Step>
  <Step title="Approval requests avoid replayable token values">
    `approvalRequest` records must not contain token values that could be replayed. Only binding metadata is permitted.
  </Step>
  <Step title="Secret references carry only reference metadata">
    `secretRef` records carry scoped reference metadata and redaction state. Actual secret values must never appear in a `secretRef`.
  </Step>
  <Step title="Evidence packets reject secret-bearing packets without redaction">
    `evidencePacket` records that include secret-bearing content are rejected unless a redaction policy is present.
  </Step>
  <Step title="Tool executions carry full envelope references">
    Every `toolExecution` record must include references to policy, audit, trace, evidence, artifact, and redaction outputs. Incomplete envelopes are rejected.
  </Step>
</Steps>

<Warning>
  Negative fixture coverage is required. The validation gate explicitly checks that denied actions without denial evidence, renderer errors without error state, and unsafe UI props all fail validation. If you add a new anchor, you must supply both positive and negative fixtures.
</Warning>

## Fixture Coverage

Compatibility fixtures in `fixtures/compatibility/` and `fixtures/shared-seams/` prove every anchor against the contract gate. Coverage spans:

- Unsupported UI components and invalid payloads
- Denied actions, renderer error states, and artifact views
- Prompt injection denial, tool metadata poisoning denial, and MCP transport abuse denial
- Secret exfiltration denial and cross-scope action denial
- Approval replay rejection and secret-reference value leakage rejection
- Completed, denied, unsupported, and invalid tool execution records
- Shared seam coverage for `runEvent`, `uiPayload`, `artifactView`, `actionRef`, `themeRef`, `suiteRef`, `evidencePacket`, `memoryRecord`, `contextPack`, and `capabilityManifest`

## Schema Validation

The package ships a validation gate that enforces fixture coverage and cross-field contract semantics on every schema. When you consume the published package, the generated artifacts are already validated — no additional steps are needed on your end.

If you are contributing schema changes to the harness repository, run the full validation suite after modifying any schema file to confirm generation, drift, and fixture coverage all pass. Refer to the contributor guide in the repository for details.
